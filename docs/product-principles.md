# 제품 및 사용자 답변 원칙

## 해결하려는 문제

일반 지하철 길찾기는 최단 시간과 환승 횟수를 알려주지만, 휠체어·유모차·보행약자가
실제로 이동할 때 필요한 엘리베이터 위치, 운행 상태, 환승 동선, 출구 동선,
장애인화장실 정보를 한 번에 판단하기 어렵다.

Barrier-Free Mobility MCP는 여러 공공데이터를 정규화하고 사용자의 이동 조건을 반영해
다음 세 가지를 구분해서 제공한다.

1. 현재 데이터로 확인된 사실
2. 데이터만으로 확인하지 못한 동선과 시설
3. 출발 전에 사용자가 직접 확인해야 할 행동

이 서버는 일반 길찾기 앱을 대체하지 않으며, 현장 안전을 보장하지 않는다.

## 핵심 사용자 질문

- 휠체어 또는 유모차로 이 경로를 이용할 때 무엇을 확인해야 하는가?
- 출발역·환승역·도착역의 엘리베이터 위치와 상태가 확인됐는가?
- 승강장부터 대합실, 환승 통로, 출구까지 필요한 동선이 확인됐는가?
- 필요한 위치에 장애인화장실이 확인됐는가?
- 공공데이터가 부족하거나 실패했다면 정확히 어느 역과 정보가 미확인인가?

## 답변 우선순위

`user_message`는 다음 순서로 정보를 제공한다.

1. **현재 결론**: 지금 데이터로 판단 가능한 범위와 가장 중요한 제약
2. **바로 할 일**: 출발 전에 확인할 역·시설·질문
3. **역별 근거**: 출발역, 필요한 환승역, 도착역의 확인·미확인 정보
4. **데이터 기준**: 조회 시각과 사용한 정보 종류
5. **짧은 주의사항**: 현장 상태가 바뀔 수 있다는 한계

이동 시간, 전체 경유역, 사용자가 이미 말한 이동 조건은 판단에 필요할 때만 표시한다.
단순히 `출발역 -> 도착역`을 반복하는 추천 경로는 제공하지 않는다.

## 표현 기준

- `안전하게 이동 가능합니다`, `문제 없습니다`, `반드시 이용 가능합니다`처럼 안전을
  보장하는 표현을 사용하지 않는다.
- 엘리베이터가 역에 존재한다는 사실과 필요한 전체 동선이 연결된다는 사실을 구분한다.
- API 조회 성공 후 빈 결과와 API 실패를 같은 의미로 표현하지 않는다.
- `일부 역` 대신 확인 가능한 역명을 직접 표시한다.
- 첫 화면에서 결론과 행동을 이해할 수 있도록 짧은 문장과 목록을 우선한다.
- 표는 출발·환승·도착 핵심역을 빠르게 비교할 때만 사용한다. 복잡한 셀, 색상 의존,
  장식용 아이콘을 피하고 화면 낭독기로도 의미가 이어지게 작성한다.
- 모호한 입력에는 한 번에 하나의 구체적인 추가 질문을 한다.

## 구조화 결과와 LLM의 역할

- 접근성 판단, 위험도, 출처 상태는 서버의 deterministic rule과 Pydantic schema가 만든다.
- 서버 내부에서 LLM을 호출하지 않는다.
- MCP client의 LLM은 `user_message`를 우선 사용하고, 구조화 필드는 검증과 추가 설명에
  사용한다.
- MCP 서버는 외부 LLM의 최종 문장을 완전히 강제할 수 없으므로 tool description,
  prompt, resource, schema description에 같은 답변 정책을 유지한다.

## 기능 추가 기준

새 tool, API, schema 필드, 운영 인프라는 아래 조건을 모두 만족할 때만 추가한다.

1. 해결할 사용자 문제 또는 운영 문제가 구체적이다.
2. 기존 기능으로 해결할 수 없는 이유가 있다.
3. 정상, 빈 결과, 부분 실패, 전체 실패 동작을 정의할 수 있다.
4. mock fixture와 회귀 테스트를 추가할 수 있다.
5. 기본 로컬 사용을 불필요하게 복잡하게 만들지 않는다.

Redis, OIDC, 분산 rate limit은 MCP 자체의 필수 요소가 아니다. 단일 프로세스에서는
memory cache와 최소 운영 설정을 기본으로 사용하고, 다중 instance 또는 공개 운영 요구가
확인될 때만 선택한다.

## 대표 검증 시나리오

- 직행 경로에서 출발역·도착역 엘리베이터 상태는 확인됐지만 내부 동선은 미확인
- 환승 경로에서 환승역 엘리베이터 또는 환승 동선이 미확인
- 엘리베이터 점검·이용불가 시설이 포함된 경로
- 장애인화장실의 요구 위치가 확인되거나 미확인인 경로
- 호선 없는 환승역 또는 복수 역 후보를 포함한 모호한 질문
- 공공 API 일부 실패, 지원 범위 밖, stale cache 응답

변경은 위 시나리오의 fixture와 테스트를 먼저 갱신한 뒤 구현한다.
